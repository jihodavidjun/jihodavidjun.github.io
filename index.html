Projects

Selected work across physical design, secure hardware, and research.

Project 01

SRAM Integration and Physical Design for a SKY130 RISC-V SoC

Through Silicon Jackets, I worked on integrating ChipFoundry SRAM macros into a SKY130-based RISC-V SoC flow and helping push the design toward signoff and tapeout. This work required much more than simply instantiating a memory block in RTL. The challenge was to make the SRAM behave correctly inside the full backend flow by aligning logical views, physical views, power connectivity, and implementation constraints. I worked with LEF, Liberty, and GDS views, adjusted setup and flow scripts, debugged macro-related physical design issues, and iterated on power grid connectivity and timing behavior.

A major part of the project involved understanding what happens when a hard macro is introduced into an otherwise standard-cell-driven implementation flow. That included floorplan awareness, routing around macros, power planning, checking DRC and LVS behavior, and dealing with MMMC timing scenarios in Cadence tools. This experience gave me a much stronger appreciation for the real complexity of physical design work: backend issues are rarely isolated, and solving one problem often changes the behavior of power, routing, or signoff elsewhere in the flow.

Cadence Innovus | SKY130 | MMMC | DRC/LVS | SRAM Integration

Project 02

AES-128 Secure Voting Machine Integration and Verification

In Georgia Tech’s VIP Secure Voting Machine project, I worked on integrating an AES-128 accelerator into a dual-core RISC-V system. This project connected several layers of hardware engineering: RTL design, interface logic, system integration, and verification. The AES engine was wrapped with AXI4-Lite so it could be controlled by processors in the system, which meant thinking carefully about memory-mapped control, data movement, and the interaction between hardware and software.

I also worked on validation and verification infrastructure. That included self-checking tests using NIST Known Answer Tests and Monte Carlo style repeated testing, along with firmware-based validation through UART and memory-mapped I/O. I liked this project because it showed how verification is not a separate afterthought; it is part of making a system trustworthy. Beyond getting the block to function, the goal was to create a reliable path for proving that it functioned correctly under repeated use and in a realistic SoC environment.

Verilog | AXI4-Lite | RISC-V | Firmware Validation | AES | Verification

Project 03

Hardware Semantic Embeddings Research at SHARC Lab

In research, I have been exploring the relationship between high-level hardware specifications and lower-level implementations, especially in HLS-driven design flows. The central question is whether machine learning models can learn useful semantic representations of hardware designs that reflect how closely an implementation matches its specification. To study this, I have worked with pipelines that materialize many HLS design examples and variants, organize them into datasets, and prepare them for encoder-based analysis.

This project interests me because it sits directly between hardware engineering and ML for EDA. It is not only about building models, but also about defining what the data should be, what counts as a meaningful representation of a hardware design, and how to evaluate whether an embedding is useful. Working on this has made me think more deeply about design intent, implementation detail, and how tooling could eventually help engineers reason about large hardware design spaces more efficiently.

HLS | ML for EDA | Design Representations | Python | PyTorch

Project 04

Systolic Array GEMM Accelerator

I designed and studied a systolic array accelerator for matrix multiplication, with a 4×4 processing structure and a scalable path toward larger NxN designs. This project was valuable because it connected computer architecture ideas with implementation realities. On paper, systolic arrays are elegant and highly parallel, but actual performance depends on how data is fed, buffered, synchronized, and reused across the array.

I analyzed performance bottlenecks and compared behavior against software-based models, including PyTorch references. One of the most useful takeaways from this work was seeing how quickly theoretical throughput can fall when the memory system becomes the limiting factor. That reinforced a lesson I keep seeing across hardware projects: design quality is not just about the compute structure itself, but about how the whole data path is organized around it.

Computer Architecture | Accelerator Design | Streaming Dataflow | PyTorch
